OAuth 2.0

OAuth 2.0 is now available for selected REST API clients. OAuth is "an open protocol to allow secure API authorization in a simple and standard method from desktop and web applications". Socialcast added this functionality to remove the need for third party clients to securely store your credentials. Instead of storing an encrypted copy of your password, the OAuth enabled client applications now use an access token to authenticate with Socialcast. This token can be used to perform a limited set of actions your behalf, but doesn't permanently provide full access to your account. In the event that an authorized application is compromised, attackers can't access the user's plain text password and the compromised tokens can be revoked at any time.

To register a custom app that a developer has created for your community, please refer to the documentation for the Community Apps Manager.

Client Registration

Socialcast requires all clients to be registered with Socialcast before they can use the OAuth2 authentication services. This process is currently manual. Please contact support@socialcast.com with the following details to request client registration:

  • Email subject: OAuth2 Client Registration Request for [Application Name]
  • Application name: The name of the application that will be displayed to users
  • Application description: A short description of the application
  • Application homepage URL: A URL with more information about the application.
  • Application callback URL: A URL that will be used to pass data back to your application via user-agent redirection. The Callback URL must use SSL.

Upon successful registration, the you will receive your the following information from Socialcast:

  • client_id: an opaque string that uniquely identifies your client application to Socialcast
  • client_secret: an opaque string associated with your client id, and used to authenticate requests as having originated from your application (not necessary in all flows)

Determining Community Domain

By default, API clients should present the user with the centralized authorization endpoint at:

https://api.socialcast.com/oauth2/authorization

This enhanced authorization endpoint allows many users to log in to any of their Socialcast communities, provided these communities:

  • Are Standard SaaS accounts (i.e. not Private Cloud or On Behind the Firewall)
  • Do not use SSO (single sign on) to authenticate users

Upon successful authorization, Socialcast will append a "community_domain" argument to the OAuth 2 authorization response (see Authorization Code and Implicit grant types) containing the domain you can use to redeem an authorization token (see the Authorization Code grant type) and make other API requests (see Accessing Protected Resources).

Other Cases

Unfortunately proper delegation to other identity providers and integration with private cloud or behind the firewall deployments requires API clients to use the full community domain when accessing OAuth 2.0 authorization endpoint. This is the same domain that would appear in a browser's URL bar after you have logged in to a community. To help verify that a particular domain is a valid communty domain, OAuth 2.0 compatible versions of Socialcast provide a "check" method that does not require prior authorization at:

http://COMMUNITY_DOMAIN/oauth2/check

For example, the following request:

GET /oauth2/check HTTP/1.1
Host: demo.socialcast.local
Accept: */*

will return something similar to the following JSON:

{
  "tenant": "Foobar Inc",
  "service_type": "Socialcast",
  "protocol_type": "OAuth2"
}

Note: the value of "tenant" will be null if the server is OAuth2 compatible, but the domain provided does not correspond to a tenant.

Supported Flows

Authorization Code Grant - See Section 4.1 of Draft 16. This flow is suitable for applications which can make cross-domain HTTP requests, such as web, mobile, or desktop applications.

Implicit - See Section 4.2 of Draft 16. This flow is recommended for clients which cannot make cross-domain requests or keep secrets such as in-browser JavaScript applications.
